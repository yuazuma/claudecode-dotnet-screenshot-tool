# v1.1.0 設計書更新の思考記録

## 更新方針の判断

### 設計書の粒度バランス

基本設計書・詳細設計書・導入手順書の3文書をそれぞれ異なる深さで更新した。

- **基本設計書**: アーキテクチャ・クラス設計・データモデル・処理フローを追加。コード断片は「フロー説明用の擬似コード」程度。
- **詳細設計書**: 実装レベルの詳細（COM GUID、メソッドシグネチャ、バイナリ構造等）を追記。コードも具体的なC#サンプル付き。
- **導入手順書**: ユーザー向け設定項目の説明に特化。内部実装への言及は最小限。

この粒度分けは元の文書構成の意図と一致しており、設計書を読む対象者（実装者 vs ユーザー）に合わせた。

### MfVideoWriter の COM GUID の正記載

詳細設計書 3.26 に `IMFSinkWriter` の正しい GUID `{3137f1cd-fe5e-4805-a5d8-fb477448cb3d}` を明記した。
前セッションで誤った GUID が原因で MP4 が 0 バイトになるバグを検出・修正した経緯があり、
「なぜこの値なのか」が明確でないと将来また間違えるリスクがある。
他の COM インターフェース GUID も一覧として記載することで、将来の修正時の比較対象にできる。

### VideoGenConfig の JSON キーマッピング

C# プロパティ名（PascalCase）と JSON キー名（camelCase）の変換については
`System.Text.Json` の既定動作に依存しているが、これを詳細設計書に明記した。
理由: 手動で config.json を編集する際の混乱防止と、将来シリアライズ設定を変更した場合の影響範囲の明確化のため。

### セクション番号の追加戦略

既存のセクション番号（3.1〜3.21, 4.1〜4.3 等）を変更せず、末尾に 3.22〜3.27, 4.4 として追加した。
途中に挿入すると既存の参照（他文書・コメント等）が壊れるリスクがあるため末尾追加を選択。

### CHANGELOG の形式選択

Keep a Changelog 形式（keepachangelog.com）を採用した。
理由:
- Added / Changed / Fixed の3カテゴリが今回の変更内容（新機能・バージョン更新・バグ修正）と自然に対応する
- 標準的なフォーマットなので読み手が構造を予測しやすい
- セマンティックバージョニングとの親和性が高い

### リリースノートの言語分離

JA/EN を別ファイルにした。v1.0.0 の既存ファイル（RELEASE_NOTES_JA.md / EN.md）は
バージョン番号なしで1つのファイルに全バージョンが積み重なる想定だったかもしれないが、
バージョン別ファイル（RELEASE_NOTES_v1.1.0_JA.md）に切り替えた。
理由: 複数バージョンのリリースノートを1ファイルにまとめると
「最新版だけ確認したい」ユーザーにとって読みにくくなるため。

## 各クラスの設計記述で意識したこと

### ApngWriter のバイナリ構造

APNG の仕様はブラウザ実装によって解釈が異なる細かい点がある。
特に「第1フレームを IDAT で書く vs fdAT で書く」という点は、
APNG の仕様（第1フレームは IDAT + PNG互換）に従い IDAT で書くことを明記した。
これにより APNG 非対応ビューアでも第1フレームが静止画として表示される。

### TtsService の STA スレッド問題

`SpeechSynthesizer` は COM ベースで STA スレッドを要求するケースがある。
`Task.Run` は MTA スレッドプールを使うため、理論的には問題になる可能性がある。
ただし実際の動作確認（前セッション）では問題なく動作したため、
詳細設計書には「MTA でも動作する場合は特別な対応不要」として注記にとどめた。
将来問題が発生した場合のヒントとして残す意図。

### FrameRenderer の WebP 対応

`step.ImagePath` が `.webp` の場合の処理を詳細設計書に明記した。
GDI+ が WebP を直接読めないため `SixLabors.ImageSharp` 経由で PNG に変換する。
ただしこの変換は同セッションで `FrameRenderer.cs` に実装済みであり、
設計書は実装の後追い記述になっている。
（通常は設計→実装の順だが、今回は実装→設計書更新の順序）
