# v1.7.0 実装の思考

## FR-H1+H2: ロスレス保存の設計判断

### WebP の lossless 設定

`WebpEncoder.Lossless = true` と書いたが、SixLabors.ImageSharp 3.1.7 の API では
`Lossless` という bool プロパティは存在せず、
`FileFormat = WebpFileFormatType.Lossless` が正しい。

これは SixLabors.ImageSharp のバージョンによって API が変わっていることによる。
ビルドエラーで判明し修正した。

### `lossless` パラメーターを CaptureService 側に閉じた理由

FR-H1（before ロスレス）と FR-H2（after ロスレス）は本質的に同じ処理。
`CaptureService.Encode(bmp, format, jpegQuality, lossless: true)` という単一 API で
両方を対応できるため、TriggerOrchestrator から `lossless: true` を渡すだけで済む。

`lossless=false` のパスを残しているのはサムネイル生成（ThumbnailService）で
lossy 圧縮を継続使用するため。before/after 画像専用に lossless をハードコードしない設計。

## FR-H3: フォルダテンプレートエンジンの設計

### 2パス処理の理由

テンプレート `{date_time}_{title_short}.ascproj` を評価する際、
以下の競合問題があった:

**問題**: タイトルが "Meeting May dd" のように DateTime 書式指定子を含むと、
`DateTime.ToString()` で `dd` が日付として解釈される。

**解決**: 2パス処理
1. `{placeholder}` を **計算済み文字列** に展開（日時プレースホルダーも即評価）
2. 残った DateTime 書式トークン（`yyyy` `MM` 等）を `DateTime.ToString()` で評価

この順序では `{title}` の中身は Step 1 で文字列に確定しているため、
Step 2 の `DateTime.ToString()` で誤変換されない。

### `{date_time}` の実装

`{date_time}` は `yyyyMMdd_HHmmss` の短縮形だが、
`_` が DateTime 書式では literal 文字のため通常は問題ない。
しかし `{date_time}` を `yyyyMMdd_HHmmss` という書式文字列に展開すると、
`DateTime.ToString("yyyyMMdd_HHmmss")` が `20260530_143022` を返す。

テスト: `sessionStart.ToString("yyyyMMdd_HHmmss")` で確認済み。✓

実装では `{date_time}` をそのまま書式文字列に置換せず、
直接 `ToString("yyyyMMdd") + "_" + ToString("HHmmss")` で計算して文字列を返した。
これにより `_` の書式解釈の曖昧さを完全に排除。

### `.ascproj` 拡張子の保護

テンプレート評価後に `.ascproj` が含まれていない場合は補完する設計。
テンプレート `{date_time}_{title_short}` だけでも動作し、
`.ascproj` は必須ではないがシステムが補完する。

## FR-H4: SettingsWindow の XAML 問題

### `x:String` vs `sys:String`

ツールヒント用の長い文字列を `Window.Resources` に `x:String` として定義しようとしたが、
WPF では `x:String` は認識されない（WinUI/UWP の機能）。

修正: `xmlns:sys="clr-namespace:System;assembly=mscorlib"` を追加し
`<sys:String x:Key="...">` を使用。

また endタグも `</x:String>` → `</sys:String>` に変更が必要だった。

### VideoGenerator の `session` スコープ問題

`ResolveOutputFolder(cfg, session)` のシグネチャ変更で
`GenerateForSteps()` 内でローカル変数 `session` が存在しなかった。

`title` と `startedAt` は引数で取れているので、
`var sessionMock = new ManualSession { Title = title }` で簡易 ManualSession を生成して渡した。
実際に使われるのは `session.Title` と `session.StartedAt` のみなので問題なし。

## FR-H5: フォールバックの設計

### SaveAsync 内でフォールバックを完結させる

当初、TriggerOrchestrator や外部でフォールバックを管理することも考えた。
しかし「書き込み失敗を検知したら即座に切り替え」という要件のため、
`SaveAsync()` の try-catch の中でフォールバックへの再保存まで完了させる設計にした。

これにより:
1. 失敗した書き込み = フォールバック先への再保存 が atomic になる
2. 呼び出し元（TriggerOrchestrator）はフォールバックを意識しない

### セッション内での1回限り切り替え

`_usingFallback` フラグで「既に切り替え済み」を管理。
フォールバック先も失敗した場合は `throw` して呼び出し元にエラーを伝播。
無限ループ的なフォールバック連鎖は起きない。

## FR-H6: 進捗ダイアログの設計

### ExportProgressAdapter の役割

`IProgress<ExportProgress>` と `CancellationToken` をセットにする。
ExportService・各 Writer はこの2つを別々の引数で受け取るが、
呼び出し元（ProjectViewWindow）では1つのオブジェクトとして扱える。

### `Progress<T>` の UI スレッドディスパッチ

`new Progress<ExportProgress>(onReport)` は `Report()` を呼んだスレッドではなく、
`Progress<T>` が作成された同期コンテキスト（= UI スレッド）で `onReport` を実行する。

`ExportProgressAdapter` を UI スレッドで new するため、
`Dispatcher.BeginInvoke` なしで直接 `win.UpdateProgress(p)` を呼べる。

ただし、ProjectViewWindow の `RunExportWithProgress` 内で
`Dispatcher.BeginInvoke(() => win.UpdateProgress(p))` とした点は冗長だが、
安全のための二重確認として残した。

### VideoGenerator の `mp4Path2` 変数

MP4 フレームループに進捗報告を追加する際、ループ外の `mp4Path` 変数を
参照しようとしたが、それは既に `if (cfg.OutputMp4)` ブロック内にある変数だった。
`string mp4Path2 = Path.Combine(folder, baseName + ".mp4")` としてループ外で再定義した。
これは名前の重複を避けるための暫定対処で、本来は元の変数名が適切。
