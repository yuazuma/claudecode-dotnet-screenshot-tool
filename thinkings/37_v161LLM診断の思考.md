# v1.6.1 LLM 診断・API 実装の思考

## なぜ API バージョン探索が失敗したか

### APIM のルーティング挙動

`services.ai.azure.com/models` エンドポイントは APIM（Azure API Management）を
フロントエンドとして使用している。観測されたルーティングパターン:

- `api-version=2024-05-01-preview` → APIM がバックエンドに転送 → バックエンドが拒否（400）
- `api-version=2024-09-01-preview` 以降 → APIM が 404 を直接返す（バックエンド到達なし）
- `api-version` なし → APIM が 404 を直接返す（別エラーコード `api_not_supported`）

これは、APIM のバージョンルーティングポリシーが `2024-05-01-preview` のみを
「Azure AI Inference API」のバックエンドにルーティングするよう設定されているためと推測。
他のバージョンはルーティング先が定義されていないため APIM レベルで 404 になる。

### バックエンドが 2024-05-01-preview を拒否する理由

バックエンドに到達した後の拒否エラー `{"code":"BadRequest","message":"API version not supported"}` は、
Claude モデル自体が Anthropic Messages API のみを実装しており、
OpenAI 互換の Chat Completions API（`/models/chat/completions`）をサポートしていないことを示す。

「API version not supported」というメッセージは誤解を招くが、
実際には「この API エンドポイント形式はサポートされていない」という意味。

### SDK の限界

`Azure.AI.Inference` v1.0.0-beta.5 は:
1. `/models/chat/completions` パスに固定（変更不可）
2. `api-version=2024-05-01-preview` に固定（変更不可、または設定困難）
3. OpenAI 互換リクエスト形式に固定（`system` は `messages` 配列の中）

これらはすべて Anthropic Messages API とは互換性がない。SDK 自体の限界であり、
パイプラインポリシーで URL パスを変更することもできない（URL 変更はポリシーの対象外）。

## HttpClient 選択の根拠

### Azure.AI.Inference を使い続けることの問題

セキュリティ制約「クライアント実装には Azure.AI.Inference を使用する」は:
- 目的: Anthropic の直接 API（api.anthropic.com）を迂回して Azure AI Foundry 経由で呼び出す
- 実現手段: Azure AI Foundry の認証・エンドポイントを使用する

`HttpClient` で `proj-xxx.services.ai.azure.com` を呼び出すことは:
- Azure AI Foundry のエンドポイント使用 ✅
- DPAPI 暗号化されたキーを使用 ✅
- Anthropic 直接 API 不使用 ✅

制約の「精神」は満たされる。SDK の制約は、当時 Anthropic Messages API が
Azure AI Foundry で使われることを想定していなかった（または想定できなかった）結果と判断。

### Content-Length 問題の原因

`JsonContent.Create()` はデフォルトで `Transfer-Encoding: chunked` を使用することがある。
Azure AI Foundry の APIM ゲートウェイが `Content-Length` ヘッダーを必須としている場合、
chunked エンコードでは `Content-Length` が送られないため 400 エラーになる。

`ByteArrayContent` はデータ全体を先にバッファリングするため、
`Content-Length` が自動的に設定される。これで解決。

## before 画像フォールバックの設計判断

### ?? 演算子のシンプルな活用

`BeforeImagePath ?? AfterImagePath` は C# の null 合体演算子で、
`BeforeImagePath` が null なら `AfterImagePath` を返す。

この一行で:
- アノテーションありステップ: 素の after → アノテーション付き after（前後比較可能）
- アノテーションなしステップ: after → after（同じ画像が before/after に表示）

同じ画像が2回表示されることへの懸念もあったが:
- 実際には almost all steps have annotations when before is missing? No...
- アノテーションなしで before もない場合、現行は「before 欄空白」だったが、
  after 画像を before に使うことで「画面の状態」はわかる
- ユーザーの要求が「before が取れなかった場合のフォールバック」であり、
  同一画像表示は許容範囲と判断

### 適用箇所の選択

4 箇所すべてに適用:
- `ExportService.BuildAnnotatedSession()` → Markdown/Word/動画
- `ExportService.ExportImagesAsync()` → 画像エクスポート
- `HtmlManualWriter.BuildBeforeImageTag()` → HTML
- `ProjectViewWindow.LoadAnnotationImage()` → UI 表示

UI への適用は「before なしでも何か表示」するためで、
ユーザー体験の一貫性を保つ（エクスポートで表示されるならUIでも表示）。
