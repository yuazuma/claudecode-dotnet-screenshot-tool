# プロジェクトファイル機能 要件定義の思考記録

## フォルダ形式を選択した理由

ユーザーが①フォルダ形式を選択した背景として、以下が考えられる:
- ZIP は更新ごとに再圧縮が必要で、高頻度撮影（秒単位）には不向き
- バイナリ独自形式はサードパーティツールで開けず可搬性が低い
- フォルダ形式は Windows Explorer でそのまま閲覧でき、誤って破損した場合も個別ファイルを直接修正できる

`.ascproj` という拡張子は、「AutoScreenshot Project」の頭文字をとったもの。
フォルダに拡張子を付けることで、スクリプトやツールでのフィルタリング（`.ascproj` だけ列挙等）が容易になる。

## project.json の書き込み方式

高頻度撮影時（連続クリック等で数百ms間隔）に project.json への書き込みが競合するリスクがある。
対策として:
1. `SemaphoreSlim(1,1)` or `lock` で排他制御
2. `.tmp` ファイルに書いてからリネーム（原子的置換）で途中終了時のファイル破損を防ぐ

全体書き直し（上書き）方式を選択した理由:
- 追記のみでは steps 配列の中間削除（isDeleted 設定）が難しい
- JSON マージパッチは実装が複雑
- 1000 ステップで約 500 KB — 毎回全体書き直しでもパフォーマンス上の問題はない

## サムネイル生成の判断

サムネイルを `thumbs/` サブフォルダに別置きにした理由:
- プロジェクトビューの一覧表示時に `images/` の原寸ファイルを読み込むと重い（1 枚数MB×数十枚）
- JPEG Q=75 / max 320px ならば 1 枚 30KB 程度で高速に表示できる
- プロジェクトビューの GridView は `thumbs/` から読み込み、詳細表示のみ `images/` を参照

サムネイル生成の失敗は撮影継続に影響させない（try-catch で握り潰す）。
UI がサムネイルを表示できない場合は代替アイコンを表示。

## descriptionOverride の採用

説明文の優先順位を `descriptionOverride > descriptionLlm > descriptionRuleBased` とした。
- LLM 呼び出しはコストとレイテンシがある → 結果を project.json に保存（再実行不要）
- ユーザーが手修正した場合はその内容を最優先にする（LLM の改善を上書きしない）
- 空文字列を入力すると null に戻す（LLM フォールバックを復元できる）

## isDeleted とファイル移動の方式

物理削除ではなく `images/_deleted/` 移動を採用:
- 誤削除のリスク軽減（ゴミ箱相当の安全網）
- v1.2 スコープでは「元に戻す」UIは提供しないが、将来対応が可能
- ディスク容量が問題になるケースは少ない（手順書用の画像は数百MB程度）

ステップ番号（stepNumber）は削除後も変わらない。
エクスポート時は `isDeleted=false` のステップを再番号付けして出力する。

## トレイメニューの設計判断

旧メニュー（「手順書セッション区切り」「手順書を今すぐ生成」「動画を生成」）を廃止し、
「プロジェクト区切り」「エクスポート >（サブメニュー）」「プロジェクトを管理...」に統一した。

理由:
- v1.2 から「手順書」と「動画」が両方ともエクスポート操作になるため、
  個別メニュー項目として並べるより「エクスポート >」のサブメニューにまとめる方が意味論的に正確
- 「プロジェクト区切り」の名称で「セッション」から「プロジェクト」への概念変更を明示

## 設定フォールバックモードの必要性

「プロジェクト機能を有効にする」設定をオフにすると v1.1.0 相当の動作に戻る。
この設定が必要な理由:
- 既存ユーザーが v1.2 へのアップデート時に動作変更に戸惑うリスクを軽減
- プロジェクト機能が不要なライトユーザーへの配慮
- ただし将来バージョン（v1.3 等）で廃止予定とし、技術的負債の蓄積を避ける

既定値は「有効」にする（v1.2 の主目的がプロジェクト機能のため）。

## ExportService の設計方針

エクスポート操作（画像/手順書/動画/ZIP）を `ExportService` に集約した理由:
- 各エクスポートで共通する処理（isDeleted フィルタ、exportHistory への記録、UI のロック）を一か所に集める
- トレイメニュー・ProjectViewWindow の双方から同じメソッドを呼べる

`VideoGenerator` の入力を `ManualSession` から `ProjectInfo` に変更する際、
既存の `GenerateAsync()` インターフェースを最小限の変更で対応するため、
`ExportService` が `ProjectInfo` → `VideoGenerator` が受け入れる形式への変換を担う。

## ファイル関連付けを採用しなかった理由

ヒアリング第1回で「ダブルクリックで開く」という回答があったが、第2回で「ファイル関連付けなし」を選択。
この矛盾の解釈:
- フォルダをダブルクリックすると OS がそのまま Explorer で開く（フォルダの通常動作）
- 独自アプリでフォルダを開くにはファイル関連付けが必要だが、フォルダへのファイル関連付けは OS 的に複雑
- 「ファイル関連付けなし」の明示的な選択を優先し、プロジェクトビューはトレイメニューから開く設計とした

## AutoExportOnClose の既定値

終了時自動エクスポートの既定値:
- Markdown 手順書: **オン**（v1.1.0 の「終了時に自動生成」と同等の挙動を維持）
- Word: **オフ**（追加設定が必要、既定で生成する理由がない）
- 動画: **オフ**（生成時間が長く、ユーザーが明示的に要求した場合のみ実行する方が適切）

この選択により、v1.2.0 にアップデートしたユーザーが「手順書が自動生成されなくなった」と混乱することを防ぐ。
