### 質問の優先順位

機能の骨格に関わる質問を先に、詳細・オプションを後回しにする構造とした。

4. **出力形式・章立て・スクリーンショット** → ドキュメント構成の核

6. **表紙・キーボード記載・テンプレート** → 見た目・品質

## 技術選択の検討

### UI 情報取得: UIA vs OCR

**UIA (Windows UI Automation)**:
- 長所: 正確なコントロール名・型・AutomationId が取得できる
- 短所: UIA 非対応アプリ（レガシー Win32、DirectX ゲーム等）では取得不可
- .NET からは `System.Windows.Automation` 名前空間で利用可能

**OCR (Windows.Media.Ocr)**:
- 長所: ネイティブ WinRT API、追加インストール不要
- 短所: 精度がフォント・解像度に依存。小さい文字や特殊フォントは誤認識
- WinRT インターフェースを .NET から使う場合 `CsWinRT` または `Target.Windows` が必要

**組み合わせ戦略**:
1. UIA で取得試行 (タイムアウト: 200ms)
2. 失敗 → クリック周辺 ±40px の切り出し画像で OCR
3. OCR も失敗 → 座標のみ記載 + `<!-- TODO -->` マーク

### docx 生成ライブラリ

**DocumentFormat.OpenXml**（Microsoft 公式）:
- 長所: 公式・安定・Apache ライセンス・Office との高い互換性
- 短所: 低レベル API でコードが冗長になりやすい
- 目次（TOC フィールド）もサポート

**ClosedXML**・**NPOI**: 表計算向けが強く、文書作成では OpenXml に劣る

→ `DocumentFormat.OpenXml` を選択。表紙・目次・ヘッダー/フッター・ページ番号を中程度の書式で実現する。

## アーキテクチャ設計

### 既存コードへの影響最小化

`TriggerOrchestrator.FireCapture()` に直接手順書ロジックを書くのではなく、`ManualSessionRecorder` という独立したクラスを作成してサブスクライバーとして接続する方針とした。

```
TriggerOrchestrator.FireCapture()
  └→ ManualSessionRecorder.OnStepRecorded(evt, imagePath)
        └→ UiaService.GetElementAt(cursorPos)
        └→ RuleBasedDescriber.Describe(step)
```

`ManualSession` / `ManualStep` はデータモデルとして独立。生成タイミングで `MarkdownWriter` / `DocxWriter` に渡す。

### LLM 連携を後付けできる設計

フェーズ1はルールベースのみ。LLM は `IStepDescriber` インターフェースの実装として後から差し込む。

```csharp
interface IStepDescriber {
    Task<string> DescribeAsync(ManualStep step);
}
class RuleBasedDescriber : IStepDescriber { ... }   // フェーズ1
class LlmDescriber : IStepDescriber { ... }          // フェーズ4
```

### Microsoft Azure AI Foundry への変更（後続プロンプトで確定）

当初は Anthropic Claude API を直接呼ぶ設計だったが、ユーザー指示により **Azure AI Foundry 経由**に変更。

**理由**: 組織内での API キー管理・コスト管理・アクセス制御を Azure で一元化するため。

**実装の変更点**:
- `HttpClient` + Anthropic REST API → `Azure.AI.Inference.ChatCompletionsClient` + `AzureKeyCredential`
- API キー1つ → エンドポイント URL + API キーの2要素認証
- モデル ID → Foundry デプロイメント名

**DPAPI 暗号化**: `System.Security.Cryptography.ProtectedData.Protect()` で config.json に暗号化保存。ユーザーセッションスコープ（`DataProtectionScope.CurrentUser`）で保護。

## 要件の難易度評価

| フェーズ | 難易度 | 主な理由 |
|---|---|---|
| フェーズ1（基盤） | 中 | セッション管理・ルールベース・Markdown 出力は直接的 |
| フェーズ2（UIA/OCR） | 高 | UIA の非同期・タイムアウト処理が複雑。WinRT API の .NET インターフェースが煩雑 |
| フェーズ3（docx） | 中〜高 | OpenXml の低レベル API。目次フィールドの更新が Word 開き時に必要 |
| フェーズ4（LLM） | 低〜中 | Azure AI Inference SDK は簡潔。プロンプト設計に注意が必要 |
