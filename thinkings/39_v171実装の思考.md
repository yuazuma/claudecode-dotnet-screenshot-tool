# v1.7.1 実装の思考プロセス

## 1. MP4 エクスポート問題

### 問題の本質

Azure Windows Server 2025 での H.264 エンコーダー不在：
- Desktop Experience や Media Feature Pack が未インストール
- `MFTEnum(MFT_CATEGORY_VIDEO_ENCODER, ...)` が count=0 を返す
- IMFSinkWriter.Finalize() も HRESULT=0xC00D4A44 で失敗（MPEG-4 muxer 側の問題）

### 解決策の選択

**検討した選択肢:**
1. Windows.Graphics.Capture API でのキャプチャ（映像ではなく画面キャプチャ）
2. MPEG-4 Part 2 エンコーダーの利用 → Azure Server に存在しない
3. OpenH264 (Cisco) DLL のバンドル → ライセンス・配布の複雑さ
4. FFmpeg バンドル → GPL ライセンス問題、ファイルサイズ大
5. **FFmpeg を外部ツールとして使用（winget install で導入）** → 採用

**3段階フォールバック採用理由:**
- Windows クライアントでは Step 1（MfVideoWriter）が動作 → パフォーマンス優先
- Azure Server では Step 3（FFmpeg）が確実 → 互換性確保
- Step 2（H264Mp4Writer）は中間環境向け保険

### AVI廃止の判断

要件明確:「AVI形式ではなくMP4形式で出力することが必須」
→ AVI フォールバックは要件違反。コードを残すが VideoGenerator からの呼び出しを削除。

### dur バグの発見

`CalcFrameDuration()` が秒単位を返すにもかかわらず `dur / 10_000_000.0` という
100ナノ秒換算の除算が残っていた。これはフレーム時間が微小になる致命的バグ。
ソースをよく読まず「100ナノ秒単位だろう」と先入観で実装した初期ミス。

## 2. RDP キャプチャ問題

### 調査方法

まず診断スクリプトで現状確認:
- `SystemInformation.TerminalServerSession` → RDP 確認
- GDI CopyFromScreen → RGB(24,24,24)、distinct=23 → 単色に近い
- `GraphicsCaptureSession.IsSupported()` → True

**distinct=23 の意味:** 統計的にほぼ単色（1色なら 1 であるべき）だが、
アンチエイリアスやフォント等で若干の変化がある程度。実用的な画面内容ではない。

### WGC 実装での試行錯誤

**ハマりポイント1: `Direct3D11CaptureFramePool.Create` 問題**
最初に `Create` を使ったところ `FrameArrived` が 3 秒タイムアウトしても一切発火しなかった。
原因: `Create` はディスパッチャースレッドで発火するが、`Task.Run` のスレッドプールには
ディスパッチャーがない → `CreateFreeThreaded` に変更して解決。

**ハマりポイント2: IMemoryBufferByteAccess の QI 失敗**
`Marshal.GetIUnknownForObject(ref_)` + `Marshal.QueryInterface` で QI したが
`IMemoryBufferByteAccess QI 失敗` が発生。
CsWinRT (.NET 8) のオブジェクトモデルでは純粋 COM QI が CsWinRT ラッパー越しに
機能しない場合がある。
→ `SoftwareBitmap.CreateCopyFromSurfaceAsync` + `BitmapEncoder(BMP)` 経由で解決。
  コード量は増えるが確実動作。

**ハマりポイント3: UnmanagedType.HString 非対応**
`RoGetActivationFactory` の `[MarshalAs(UnmanagedType.HString)]` で `MarshalDirectiveException`。
.NET 8 P/Invoke では `HString` のマーシャリングが非対応。
→ `WindowsCreateString` / `WindowsDeleteString` で手動管理。

**ハマりポイント4: MemoryStream 解放後の Bitmap**
`new Bitmap(memoryStream)` で作成した Bitmap は MemoryStream を内部で保持するため、
`using var ms` でクローズすると後続の `Save()` で GDI+ エラー。
→ `new Bitmap(tmp)` で独立コピーを作成。

### 黄色ボーダー対策

`IsBorderRequired` プロパティは TFM 17763 の型定義に存在しない。
コンパイル時 `CS1061` エラーになる。

解決:
1. `IGraphicsCaptureSession3` を `[ComImport, InterfaceIsIInspectable]` で独自定義
2. `(object)session is IGraphicsCaptureSession3 s3` で実行時 QI（コンパイル時チェックは
   `(object)` で迂回）
3. QI 失敗時（非対応 OS）: vtable 直接呼び出しでフォールバック

CsWinRT WinRT オブジェクトへの COM QI が直接キャストで動くかどうかが不確か。
vtable フォールバックは確実だが、どちらが先に成功するかは OS バージョン次第。

## 3. Markdown 画像パス問題

### 問題の本質

MD ファイル: `exports/{timestamp}_{slug}.md`
画像ファイル: `images/step_001.png`（プロジェクトルート直下）

`Path.GetRelativePath("exports/", "images/step_001.png")` → `"../images/step_001.png"`

VS Code・GitHub の Markdown レンダラーは `..` を含む画像パスをセキュリティポリシーで
ブロックする（外部フォルダへの参照を許可しない）。

### 解決策

MD ファイルと同階層に `_images/` サブフォルダを作成してコピー。
当初は `SameFile()` で重複チェックしていたが、元ファイルとリサイズ後コピーの
サイズが異なるため「別ファイル」と判定され重複コピーが発生。
→ ソースパスをキャッシュ（Dictionary）することで解決。

## 4. 設計判断のまとめ

| 課題 | 採用 | 不採用 |
|---|---|---|
| MP4 on Azure Server | FFmpeg subprocess | OpenH264 バンドル、WDDM 迂回 |
| RDP キャプチャ | Windows.Graphics.Capture | PrintWindow 合成、DXGI Desktop Duplication |
| SoftwareBitmap→Bitmap | BitmapEncoder(BMP) 経由 | IMemoryBufferByteAccess QI、vtable 直接 |
| IsBorderRequired | COM QI + vtable フォールバック | TFM アップグレード、リフレクション |
| 画像パス | _images/ コピー | 絶対パス、../パス維持 |
