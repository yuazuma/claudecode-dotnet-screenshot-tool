# v1.6.0 動作確認・不具合修正の思考

## 不具合発見の経緯

ログに `before 画像期限切れ: "Keyboard"` が出力されていた。
125 ステップ中 1 件（91%）しか before がなかった Keyboard 分が原因。

## Root Cause 分析

### タイムライン（問題ある例）

```
16:12:12.487  最初のキー → KeyboardBeforeEvent → FireBeforeCapture（Task.Run で非同期）
              ↳ キャプチャ + エンコード + SaveBeforeAsync ≈ 100-500ms
              ↳ _pendingBeforeShots[Keyboard] = (path, capturedAt≈16:12:12.6)
16:12:12〜15  継続タイプ（アイドルタイマーがリセットされ続ける）
16:12:15.032  最後のキー → アイドルタイマー 2 秒後に発火予定
16:12:17.032  タイマー発火 → TakeAccumulatedKeys → FireCapture(Keyboard)
              → TakeBeforeShot(Keyboard) を呼ぶ
              → 経過時間: 17.032 - 12.6 ≈ 4.43 秒
              → 4.43 <= 5.0 → SHOULD NOT EXPIRE
```

理論上は期限切れにならないはず。なぜ "期限切れ" ログが出たか？

### 真の原因: _inKeyboardSession が前のセッションから引き継がれていた

`_inKeyboardSession` は `TakeAccumulatedKeys()` でのみリセットされる。
ログ直前に別のキーボードセッションがあり、そのアイドルタイマーが長く設定されていた場合:

```
T₀: 前のセッション（別のタイピング） → KeyboardBeforeEvent → capturedAt = T₀
T₁: 前のセッションの after 発火前に新しいキー入力が始まった
    → _inKeyboardSession = true のまま（前セッションの after 未発火）
    → KeyboardBeforeEvent は発火しない（_inKeyboardSession がまだ true）
    → _pendingBeforeShots の中身は T₀ 時のもの（古い！）
T₂: 新しいセッション after 発火
    → TakeBeforeShot で capturedAt = T₀ の before を取得
    → T₂ - T₀ > 5.0 秒 → 期限切れ
```

### 修正方針の選択

#### 案 A: expiry を大きくする（採用）
```csharp
double maxAgeSec = trigger == TriggerType.Keyboard
    ? _config.Config.Triggers.KeyboardIdleSeconds + 10.0  // 既定 12 秒
    : 5.0;
```
- 利点: シンプル。`KeyboardIdleSeconds` が大きくなっても比例してカバー
- 欠点: 本当に古い before が使われるケースがある（実害は小さい）

#### 案 B: _inKeyboardSession 経由で有効/無効を判定
- `TakeAccumulatedKeys` が呼ばれた = セッション終了 → その直後に before が存在するなら有効
- 実装が複雑になるので不採用

#### 案 C: before を「セッション開始ではなく after タイミングの直前」に撮影
- before の本来の意味（操作前の状態）と乖離するので不採用

### なぜ `KeyboardIdleSeconds + 10` か

- `KeyboardIdleSeconds` の既定値は 2 秒、最大 10 秒
- before は最初のキーで取得される（`capturedAt = 最初のキー時刻 + 保存時間`）
- after は最後のキーから `KeyboardIdleSeconds` 後に取得される
- before → after の経過時間 = タイプ継続時間 + `KeyboardIdleSeconds`
- 通常のタイプ継続時間が 10 秒以内であれば `10 + 10 = 20 秒` でカバーできる
- 前のセッション引き継ぎケースでも同様に 20 秒あれば対応できる
- マウスの 5 秒は変更しない（DOWN → UP が必ず数秒以内）

## 検証の難しさと対策

### UIPI によるマウス注入ブロック

RDP セッション内の Python サブプロセスから `SendInput`/`SetCursorPos` が失敗する。
エラーコード 0 で `SendInput` が返す = UIPI によるブロック。

対策: 以下の代替証拠で検証した
1. ログファイル（`app-20260529.log`）の直接確認
2. ファイルシステムの `images/before/` ディレクトリと実ファイルの確認
3. `project.json` の `beforeImagePath` / `afterImagePath` フィールド確認
4. Python で before/after 画像を横並びに結合して視覚確認

### Shell_NotifyIconGetRect の活用

`Shell_NotifyIconGetRect(hWnd=0x304bc, uID=1)` で tray icon の物理座標を取得できた（1604,1140〜1644,1200）。
ただし `PostMessage(WM_TRAYMOUSEMESSAGE, 0, WM_RBUTTONUP)` はコンテキストメニューを開けなかった（UIPI）。

## 検証の結論

- 96 件の before 画像が `images/before/` に保存されていた（実在を確認）
- Keyboard 91% / MouseLeft 98% / MouseDrag 100% / MouseRight 100%
- before/after ペアの画像内容が正しく異なる状態を撮影していた
- `project.json` で旧 `imagePath` が 0 件（後方互換移行が正常）
