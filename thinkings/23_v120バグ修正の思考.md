# v1.2.0 バグ修正の思考

## 発見の経緯

v1.2.0 実装後に exe を起動して検証したところ、3つのバグを発見した。

---

## Bug 2: thumbPath のバックスラッシュ問題

### 気づき

`project.json` を確認したとき、`imagePath` は `"images/..."` なのに
`thumbPath` だけ `"thumbs\\step_001.jpg"` になっていた。

### 根本原因

```csharp
// ManualSessionRecorder.cs - 修正前
thumbRelPath = Path.Combine("thumbs", thumbFileName);
```

`imagePath` 側は以下のように明示的に変換していた：
```csharp
imageRelPath = Path.GetRelativePath(project.ProjectFolder, imagePath).Replace('\\', '/')
```

でも `thumbRelPath` は `Path.Combine` のままで `.Replace` を忘れていた。

### 修正の方針

`Path.Combine` を使う必要性がない（固定サブパス）ので、
文字列補間に変更するのが最もシンプル：

```csharp
thumbRelPath = $"thumbs/{thumbFileName}";
```

## Bug 3: MetadataLogger のログ出力先問題

### 気づき

ログファイルを見ると：
```
System.IO.IOException: The process cannot access the file
'...ascproj\images\events_2026-05-27.log'
```

ログが `images/` 配下に書かれている。

### 根本原因

```csharp
// MetadataLogger.cs - 修正前
string logDir = Path.GetDirectoryName(imagePath)!;
```

v1.1.0 まで: `imagePath = {SaveFolder}/{date}/screenshot.png`
→ `logDir = {SaveFolder}/{date}/` ← 正しい（日付フォルダ）

v1.2.0 で: `imagePath = {project}/images/screenshot.png`
→ `logDir = {project}/images/` ← 間違い（`images/` 配下）

### 修正の方針

`ResolveLogDir` ヘルパーで「images という名のディレクトリの場合は親を返す」という
プロジェクトモード対応ロジックを追加した。

この判定方法は：
- シンプル（パスの末尾ディレクトリ名を見るだけ）
- 後方互換（通常の日付フォルダは images という名前にならない）
- 堅牢（`Path.GetDirectoryName` が null を返す場合もフォールバック）

`WriteStructuredAsync` でも同じパターンを使うので、同じ `ResolveLogDir` を呼ぶように統一。

## Bug 1: project.json.tmp の残留問題

### 気づき

検証中に `.tmp` ファイルが `project.json` と同時に存在した。

`.tmp` (38895 bytes, 15:46:18) vs `project.json` (38844 bytes, 15:46:17)

サイズが違うことから、これは別々の書き込みに対応する別々のファイル。

### 分析

可能性1: 書き込み中を偶然観察した
- シーケンス: Write1完了(json=38844) → Write2開始(tmp=38895書き込み中) → 観察 → Write2のMove完了
- 3秒以上残っていたが、これはWrite2がsemaphoreを取得して完了するまでの時間かもしれない

可能性2: `File.Move` が例外をスローして `.tmp` が残留
- Windows Defender などが `project.json` を一時的にロックする可能性
- 例外は呼び出し元でキャッチされるため `.tmp` はクリーンアップされない

### 修正の方針

可能性1でも2でも、例外時に `.tmp` をクリーンアップするコードを追加することで安全になる：

```csharp
catch
{
    try { File.Delete(tmpPath); } catch { }
    throw;
}
```

`catch { }` の二重ネストは意図的：外側の catch は `.tmp` クリーンアップ、
内側の `catch { }` は削除失敗を無視（削除できなくてもスローは外側まで伝播させる）。

## 検証方法の考察

Windows Server 環境で GUI アプリを検証する際、bash からの直接起動が有効だった：

```bash
"path/to/AutoScreenshot.exe" &
```

ただし bash 経由の起動ではプロセスが見えにくいため、確認は：
- ログファイル (`app-YYYY-MM-DD.log`) を tail で確認
- プロジェクトフォルダの内容を find/ls で確認
- `tasklist` でプロセスの存在を確認
- 終了は `taskkill.exe //IM AutoScreenshot.exe //F` (bash のパス変換を避けるため `//` を使用)
