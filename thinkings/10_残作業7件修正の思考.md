## 各問題の根本原因と対処方針

### Notifier スレッドセーフ問題

**根本原因**: WinForms の `NotifyIcon` はスレッドアフィニティを持つ（STA）UI オブジェクト。`Task.Run` のバックグラウンドスレッドから直接操作すると競合状態が生じる。

**判断**: WPF アプリなので `System.Windows.Application.Current.Dispatcher.BeginInvoke()` が確実。`Control.Invoke()` (WinForms) ではなく WPF の Dispatcher を選択した理由は、Application オブジェクトが WPF 側で管理されているため。

**カウンターのスレッドセーフ化**: `lock` を使用。`Interlocked.Increment` も選択肢だったが、日付チェック（`DateTime.Today != _countDate`）との原子的操作が必要なため `lock` を採用。

---

### OnDiffTimer Error 抑制

**根本原因**: RDP 接続環境や仮想ディスプレイでは `Screen.AllScreens` が物理画面の HMONITOR を返すが、`CopyFromScreen` は実際のDC が無効なためエラー。

**NativeErrorCode == 6** は `ERROR_INVALID_HANDLE`。

**判断**: 全ての `Exception` をキャッチするのではなく、特定の Win32 エラーコードのみを `when` 節でフィルタリング。これにより本物のエラー（メモリ不足など）は引き続き `[ERR]` として記録される。ログレベルは `Debug` が適切（運用上は気にしなくていい情報）。

---

### DragThresholdMs / WheelIdleMs 保存問題

**根本原因の詳細**: `System.Text.Json` の `JsonSerializer.Deserialize<T>()` は、JSON に存在しないプロパティに対して C# のデフォルト値（プロパティ初期化子の値）を使う。ただしこれは「既存 JSON に新しい C# プロパティを追加した場合」にのみ起きる。JSON 書き直し (`Save()`) するまで新プロパティは JSON に永続化されない。

**解決策の判断**: `Load()` 直後に無条件で `Save()` する。デメリットはディスク書き込みが1回増えること（起動時のみ）。それより確実性を優先。

---

### ImageOverlay 削除

**判断**: 未実装の機能フラグを設定ファイルに残しておくと、将来的に誤って有効化される可能性がある。要件定義書に記載がない状態での dead code であるため削除が正しい。後の6件修正で再度実装することになった（設計判断として正しかった）。
