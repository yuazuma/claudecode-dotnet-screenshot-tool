# v1.4.0 アイコン状態設計の思考

## 状態機械の選択

「状態パターン」を採用するか「フラグ+優先度ロジック」にするかを検討した。
状態数が5つかつ遷移ルールがシンプル（優先度順）なため、フラグ＋`ApplyDisplayState()`の方が可読性が高いと判断。

## Captured フラッシュと Processing の競合

撮影中にエクスポートが走っていると `_capturedActive` と `_processingCount > 0` が同時に true になりうる。
優先度を `Processing > Captured` としてもよかったが、ユーザー体験上「撮れた」フィードバックは常に見せたい（200msなので邪魔にならない）ため `Captured > Processing` とした。

## ExportService への変更スコープ

- `ExportZipAsync` は UI から直接呼ばれず `ProjectViewWindow` からのみ呼ばれるため、今回は Processing 通知対象外とした。
- `ExportDocxAsync` も同様に追加した（ただし将来呼ばれる可能性があるため対応済み）。

## 一時停止中の Processing

「一時停止中にエクスポート完了 → `EndProcessing()` → `ApplyDisplayState()` → `_isPaused=true` → Paused に戻る」
という流れを確認。Notifier は `_isPaused` を内部保持しているため、NotifyIconWrapper から基本状態を再通知する必要はない。

## SetPausedState 廃止

旧 API: `SetPausedState(bool paused, NotifyIcon icon)` は icon を引数に受け取っていた（Notifier が NotifyIcon 参照を持ちながら引数でも受け取る冗長な設計）。
新 API: `SetBaseState(bool paused)` — icon は `SetNotifyIcon()` で渡し済みなので引数不要。

## Processing アイコンの視覚的区別

オレンジ背景のみでは「なにかエラー？」と誤認されるリスクがある。
右下に白い小ドット（半径 size/8）を追加して「動いている」感を表現した。
アニメーションは WinForms の NotifyIcon では非公式手段（タイマーで差し替え）が必要で、今回のスコープ外とした。
