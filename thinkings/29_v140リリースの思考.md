# v1.4.0 リリースパッケージ作成の思考

## dotnet publish vs dotnet build

リリースバイナリ作成には `dotnet publish` を使用。`dotnet build` は DLL と依存 DLL を bin/ に出力するが、
self-contained publish は OS ネイティブの .NET ランタイム DLL 一式も含めて publish_tmp/ に一括出力する。
Release 配布は常に `publish -r win-x64 --self-contained true` が必要。

## ZIP 作成の実装選択

前バージョンから踏襲: Python `zipfile` モジュールを使用。
- `powershell Compress-Archive` は PowerShell deny ルールに抵触するため不可
- `zip` コマンド（POSIX）は Git Bash 環境には存在するが、Windows パス指定と相性が悪い場合がある
- Python は確実に存在し、`hashlib` と同一スクリプトで SHA-256 も一括計算できる

## ZIP サイズの比較

| バージョン | ZIP サイズ |
|---|---|
| v1.0.0 | 86,209,829 bytes |
| v1.1.0 | 81,757,399 bytes |
| v1.2.0 | ※未記録 |
| v1.3.0 | ※未記録 |
| v1.4.0 | 81,816,195 bytes |

v1.4.0 と v1.1.0 がほぼ同サイズ（差分 58 KB）。
v1.4.0 で追加されたコードは純粋な C# / XAML であり、新規 NuGet 依存はない。
ランタイム DLL は変わらないため ZIP サイズもほぼ横ばいになるのは妥当。

## publish_tmp/ の後処理

ZIP 生成後、`publish_tmp/` は不要だが `rm -rf` が deny されたため手動削除が必要。
次回リリース時も同様の状況になる場合は、別の一時ディレクトリ名か Python の `tempfile.mkdtemp()` を検討する。

## v1.4.0 の変更がバイナリ互換か

- `config.json` 形式の変更なし: ユーザー設定はそのまま引き継げる
- `.ascproj/` フォルダ形式の変更なし: 既存プロジェクトデータはそのまま使用可能
- `Notifier.SetBaseState()` は `NotifyIconWrapper` 内部で呼ばれる API のため外部影響なし
- UI レイアウト変更は純粋な表示面のみ

以上より、アップグレードに際してデータ移行は不要。
