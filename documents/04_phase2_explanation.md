### Phase 2 でやること

**テーマ: 自動撮影 — マウス・キーボード・ウィンドウ切替の実機動作完成**

スケルトンコードで骨格はできていますが、以下の仕様が未実装または未完成です。

#### 1. F-01: マウスイベント連動撮影の完成

| 項目 | 現状 | やること |
|------|------|----------|
| 左/右/中クリック | フック設置済み・コールバック有 | 実機テストで動作確認 |
| **ドラッグ&ドロップ** | ❌ 未実装 | `WM_LBUTTONDOWN` → `WM_LBUTTONUP` の時間差でドラッグ判定、クリックと排他制御 |
| **ホイール操作** | イベント受信のみ | 最終ホイールから N ms のアイドル待機タイマー実装 (現状は即時発火) |

**ドラッグ判定の仕様:**  
マウスダウン〜アップが一定時間以上 → ドラッグとみなして `DragDrop` トリガーで撮影、クリックとしては撮影しない

#### 2. F-02: キーボードイベント連動撮影の実機確認

スケルトンでは `KeyboardIdleSeconds` (初期値 2 秒) のタイマーを使っています。Phase 1 動作確認では端末環境の制約でフックが発火しませんでした。

- 実際のデスクトップで Win32 `WH_KEYBOARD_LL` が発火するかを確認
- アイドル待機中にさらにキー入力があった場合の「タイマーリセット」動作を確認

#### 3. F-03 方式A: アクティブウィンドウ切替検知の実機確認

スケルトンでは `SetWinEventHook(EVENT_SYSTEM_FOREGROUND)` を使っています。

- 実際にウィンドウを切り替えたときに `ActiveWindowChanged` イベントが発火するかを確認
- 同一プロセス内のウィンドウ切り替え (タブ切り替えなど) でも発火するかを確認

#### 4. クールダウン・重複撮影防止の検証

各トリガー種別ごとの最低撮影間隔 (クールダウン) が、実際の操作で正しく動くかを確認します。

| トリガー | 初期値 |
|---------|--------|
| 左クリック | 1 秒 |
| ホイール | 2 秒 |
| キーボード | 2 秒 |
| ウィンドウ切替 | 1 秒 |

#### 5. Phase 1 で発見した課題の修正

- **差分タイマー例外ハンドリング** → 追加済み
- **ホイールアイドル実装** → 未実装のまま
- **ドラッグ検知** → 未実装のまま

### 実施順序 (推奨)

```
1. HookService にドラッグ&ドロップ検知を追加
   (WM_LBUTTONUP 追跡 + 時間差判定)

2. HookService にホイールアイドルタイマーを追加
   (最終ホイール後 N ms の一段階バッファ)

3. 実機 (デスクトップ) でマウス/キーボード/ウィンドウ切替を操作
   → Pictures\AutoScreenshot に PNG が保存されることを確認

4. ログで重複撮影が起きていないことを確認
   (クールダウンが機能しているか)
```

Phase 2 が完了すると、"普通にPCを使うだけで自動的にスクリーンショットが撮れる" 状態になります。
