# 追加要件定義書 — 操作手順書の自動生成機能

バージョン: 1.0  
作成日: 2026-05-26  
対象アプリ: AutoScreenshot v1.0

---

## 1. 概要

AutoScreenshot が自動収集したスクリーンショット・操作イベント・UI 要素情報を元に、**操作手順書 (Markdown / Word docx)** を自動生成する機能を追加する。  
業務 SOP 作成・社内研修資料作成・操作証跡のドキュメント化を主なユースケースとする。

---

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **セッション** | 手順書一文書の記録単位。AutoScreenshot 起動時に自動開始し、終了時に自動完了する。途中で手動区切りも可能。 |
| **ステップ** | セッション内の個々の操作記録。イベント1件（クリック等）に対応する手順書の1行または1段落。 |
| **チャプター** | アクティブウィンドウが切り替わるごとに区切られる手順書の章単位。 |
| **UI 要素** | クリック先・入力先の UI コントロール（ボタン、テキストボックス等）。UIA/OCR で取得する。 |

---

## 3. 機能要件

### 3.1 セッション管理 (S)

| ID | 要件 |
|----|------|
| **S-01** | AutoScreenshot 起動時に手順書セッションを自動開始する。 |
| **S-02** | AutoScreenshot 終了時にセッションを自動完了し、設定された出力形式で手順書を保存する。 |
| **S-03** | トレイメニューに「手順書セッション区切り」メニュー項目を追加する。クリックするとその時点でセッションを一度確定・保存し、新しいセッションを開始する（手動刺入）。 |
| **S-04** | セッション開始時に「手順書タイトル入力ダイアログ」を任意で表示できる（設定で ON/OFF）。未入力の場合は「操作手順書 YYYY-MM-DD HH:mm」を既定タイトルとする。 |
| **S-05** | セッションごとに一意なセッション ID（GUID）を付与し、ファイル名・メタデータに使用する。 |

### 3.2 UI 要素情報の取得 (U)

| ID | 要件 |
|----|------|
| **U-01** | マウスイベント（クリック・ドラッグ）発生時、クリック座標の UI 要素を **Windows UI Automation (UIA) API** で取得する。取得対象: コントロール名 (Name)、コントロール型 (ControlType)、AutomationId、ClassName。 |
| **U-02** | UIA での取得に失敗した場合、またはクリック座標の要素が UIA に対応していない場合、スクリーンショット上のクリック周辺領域（±40px）を切り出して **Windows.Media.Ocr（または Tesseract）で OCR** し、近傍テキストを UI 名として代替使用する。 |
| **U-03** | UIA・OCR ともに UI 名取得に失敗した場合、ステップを記載しつつ `<!-- TODO: UI名を確認してください -->` マーカーを埋め込む（要レビューマーク）。 |
| **U-04** | キーボードイベント発生時は、入力対象フィールドの UI 要素名を U-01 と同じ手順で取得する。 |
| **U-05** | ウィンドウ切替イベント発生時は、新しいアクティブウィンドウのタイトルとプロセス名を取得する。 |

### 3.3 操作イベントの記録 (E)

記録対象イベント：

| イベント種別 | TriggerType | 記載内容 |
|---|---|---|
| 左クリック | MouseLeftClick | UI 要素名 + 座標 + 操作目的テキスト |
| 右クリック | MouseRightClick | UI 要素名 + 座標 + 「コンテキストメニューを開きます」 |
| 中クリック | MouseMiddleClick | UI 要素名 + 座標 |
| ドラッグ&ドロップ | MouseDragDrop | 開始 UI 要素名 → 終了 UI 要素名 + 座標 |
| キーボード入力 | Keyboard | 入力先 UI 要素名 + 入力内容（設定に応じて実文字列またはキーコード） |
| ウィンドウ切替 | ActiveWindowChange | 遷移先ウィンドウタイトル + プロセス名 |

| ID | 要件 |
|----|------|
| **E-01** | 各ステップには、タイムスタンプ・トリガー種別・UI 要素名・カーソル座標・操作目的テキストを記録する。 |
| **E-02** | ステップの「操作目的テキスト」はルールベースで自動生成する（E-03 参照）。LLM が設定されている場合は LLM による上書き改善を行う（L-03 参照）。 |
| **E-03** | ルールベースの操作目的テキスト生成ルール（フェーズ1）: <br>・左クリック + ボタン → `「{UI名}」をクリックしました。`<br>・左クリック + メニュー項目 → `「{UI名}」メニューを選択しました。`<br>・左クリック + リンク → `「{UI名}」リンクをクリックしました。`<br>・左クリック + チェックボックス → `「{UI名}」チェックボックスをクリックしました。`<br>・右クリック → `「{UI名}」を右クリックしてコンテキストメニューを開きました。`<br>・キー入力（文字） → `「{UI名}」に「{入力テキスト}」と入力しました。`<br>・キー入力（Enter） → `「{UI名}」でEnterキーを押しました。`<br>・キー入力（ショートカット） → `{ショートカット名}を実行しました。`<br>・ウィンドウ切替 → `「{ウィンドウタイトル}」({プロセス名}) に切り替えました。`<br>・ドラッグ&ドロップ → `「{開始UI名}」を「{終了UI名}」へドラッグしました。` |
| **E-04** | キーボード入力の記録方式を設定で選択できる: `RealText`（実文字列記録）/ `KeyCode`（キー名のみ記録）/ `Both`（両方）。既定: `Both`。 |
| **E-05** | 画面差分検知（ScreenDiff）および手動撮影（ManualCapture）は手順書には記録しない（スクリーンショットは保存されるが、ステップには反映しない）。 |

### 3.4 手順書の文書構成 (D)

| ID | 要件 |
|----|------|
| **D-01** | **表紙**には以下を記載する: タイトル（手動入力 or 既定値）、セッション開始・終了日時、OS 名 + ユーザー名、作成ツール名とバージョン。LLM が有効な場合はダイジェスト（操作内容の要約）を追加する。 |
| **D-02** | **章（チャプター）** はアクティブウィンドウの切り替えを基準に分割する。章見出しは「{連番}. {ウィンドウタイトル}」形式とする。 |
| **D-03** | 同一チャプター内でのタイムギャップが設定値（既定: 5 分）を超えた場合、小見出し（H3 相当）で時間帯を区切る。 |
| **D-04** | 各ステップは番号付きリスト（`1.`, `2.`, ...）で記載し、チャプターをまたいでも通番を維持する。 |
| **D-05** | スクリーンショットの添付モードを設定で選択できる: `All`（全ステップ）/ `WindowChange`（ウィンドウ切替時のみ）/ `None`（画像なし）。既定: `WindowChange`。 |
| **D-06** | スクリーンショットを添付する場合、元ファイルをコピーせず相対パスで参照する（Markdown）か、docx にバイナリで埋め込む（Word）。 |
| **D-07** | **目次** を文書の先頭（表紙の後）に自動生成する。各チャプターへのリンクを含む（Markdown: アンカーリンク、Word: フィールドコードの TOC）。 |

### 3.5 出力形式 (O)

| ID | 要件 |
|----|------|
| **O-01** | **Markdown (.md)** 形式での出力をサポートする。GFM（GitHub Flavored Markdown）準拠。 |
| **O-02** | **Word (.docx)** 形式での出力をサポートする。Open XML SDK または DocumentFormat.OpenXml を使用して生成する。 |
| **O-03** | docx は以下の書式要素を含む: 表紙（タイトルスタイル）、目次（TOC フィールド）、ページ番号（フッター）、ヘッダー（文書タイトル）、手順リスト（番号付きスタイル）、画像キャプション。 |
| **O-04** | 出力ファイル名: `{セッション開始日時}_{タイトルslug}.md / .docx`（例: `20260526_143000_操作手順書.md`）。 |
| **O-05** | 出力先フォルダを設定画面から任意のパスで指定できる。既定: `{スクリーンショット保存フォルダ}/manuals/`。 |
| **O-06** | Markdown と Word の両方を同時に出力するか、どちらか一方のみかを設定で選択できる。 |
| **O-07** | ユーザーが用意した Markdown テンプレートファイル（`.md`）または Word テンプレートファイル（`.dotx`）を読み込み、スタイル・ヘッダーのカスタマイズに使用できる（任意）。 |

### 3.6 LLM 連携 (L)

> **API 呼び出し方針**: LLM の呼び出しには **Anthropic の Claude API を直接使用しない**。  
> 社内または契約環境の **Microsoft Azure AI Foundry** で払い出された API キー・エンドポイント URL を使用し、Foundry 上にデプロイされた Claude モデルを呼び出す構成とする。  
> クライアント実装には NuGet パッケージ `Azure.AI.Inference`（Azure AI Inference SDK）を使用する。

| ID | 要件 |
|----|------|
| **L-01** | 設定画面に「LLM 連携」タブを追加し、以下の入力欄を設ける（いずれもパスワード形式・非表示）: ①**Microsoft Azure AI Foundry エンドポイント URL**（例: `https://<resource>.services.ai.azure.com/models`）、②**Azure AI Foundry API キー**（Foundry プロジェクトから払い出されたキー）。 |
| **L-02** | エンドポイント URL と API キーの両方が設定されている場合のみ LLM 機能を有効化する。いずれかが未設定の場合はルールベースのみで動作する。 |
| **L-03** | **操作目的テキストの LLM 改善**: 手順書生成時（セッション完了後）に、全ステップの操作目的テキストを一括で Azure AI Foundry エンドポイント（Claude モデル）に送り、より自然な日本語説明文に改善してもらう。API 呼び出し単位はセッション全体で 1 回（コスト最小化）。 |
| **L-04** | **ダイジェスト生成**: LLM にセッション全体の操作ログ（テキストのみ）を渡し、操作内容の 3〜5 行の要約文を生成する。表紙に記載する。 |
| **L-05** | LLM への送信データにはスクリーンショット画像を含めない（コスト・プライバシーを考慮）。テキスト情報（ウィンドウタイトル、UI 名、イベント種別、入力内容）のみを送信する。 |
| **L-06** | LLM 呼び出しに失敗した場合（ネットワークエラー、API エラー、認証エラー等）、エラーをログに記録し、ルールベースのテキストで手順書を生成する（LLM なし状態にフォールバック）。 |
| **L-07** | Foundry 上のデプロイ済みモデル名（デプロイメント名）を設定で指定できる（既定: `claude-haiku-4-5`。Azure AI Foundry のデプロイ名を入力する）。 |
| **L-08** | Azure AI Inference SDK の `ChatCompletionsClient` を使用して呼び出す。エンドポイント URL と API キーを `AzureKeyCredential` として渡す構成とする。Anthropic SDK（`Anthropic.*` 系 NuGet）は使用しない。 |

### 3.7 設定 (C)

設定画面に「手順書生成」タブを追加し、以下の設定項目を提供する。

| 設定項目 | 型 | 既定値 | 説明 |
|---|---|---|---|
| `ManualGen.Enabled` | bool | true | 手順書生成機能の有効/無効 |
| `ManualGen.OutputFolder` | string | `{SaveFolder}/manuals/` | 出力先フォルダ |
| `ManualGen.OutputMarkdown` | bool | true | Markdown (.md) を出力する |
| `ManualGen.OutputDocx` | bool | true | Word (.docx) を出力する |
| `ManualGen.ScreenshotMode` | enum | WindowChange | `All` / `WindowChange` / `None` |
| `ManualGen.KeyboardMode` | enum | Both | `RealText` / `KeyCode` / `Both` |
| `ManualGen.ChapterTimeGapMinutes` | int | 5 | 同一チャプター内で時間小見出しを挿入するギャップ（分） |
| `ManualGen.ShowTitleDialogOnStart` | bool | false | セッション開始時にタイトル入力ダイアログを表示する |
| `ManualGen.TemplateMarkdownPath` | string | "" | ユーザー Markdown テンプレートファイルパス |
| `ManualGen.TemplateDotxPath` | string | "" | ユーザー Word テンプレート (.dotx) ファイルパス |
| `ManualGen.LlmEnabled` | bool | false | LLM 連携を有効にする |
| `ManualGen.LlmEndpoint` | string | "" | Azure AI Foundry エンドポイント URL（暗号化して保存） |
| `ManualGen.LlmApiKey` | string | "" | Azure AI Foundry API キー（DPAPI 暗号化して保存） |
| `ManualGen.LlmDeploymentName` | string | claude-haiku-4-5 | Foundry 上のデプロイメント名（モデル識別子） |

## 4. トレイメニューへの追加項目

| 項目名 | 動作 |
|---|---|
| 手順書セッション区切り | 現セッションを保存・完了し、新しいセッションを開始する |
| 手順書を今すぐ生成 | 現在のセッションの手順書を即時生成する（セッションは継続） |

## 5. データモデル

### 5.1 ManualSession（セッション）

```
SessionId      : Guid
Title          : string
StartedAt      : DateTime
EndedAt        : DateTime?
OsInfo         : string          // Environment.OSVersion + username
Steps          : List<ManualStep>
```

### 5.2 ManualStep（ステップ）

```
StepNumber     : int
Timestamp      : DateTime
TriggerType    : TriggerType
UiElementName  : string?         // UIA / OCR で取得
UiControlType  : string?         // Button, MenuItem, Edit, etc.
CursorPosition : Point
WindowTitle    : string
ProcessName    : string
InputText      : string?         // キーボード入力内容
KeyCodes       : string?         // キーコード列
ImagePath      : string?         // 対応スクリーンショットのパス
DescriptionRuleBased : string    // ルールベース生成テキスト
DescriptionLlm : string?         // LLM 改善テキスト（使用時のみ）
NeedsReview    : bool            // UI 名取得失敗フラグ
```

## 6. 非機能要件

| ID | 要件 |
|----|------|
| **NF-01** | 手順書生成処理はバックグラウンドスレッドで実行し、UI をブロックしない。 |
| **NF-02** | セッション完了から手順書ファイル書き出し完了まで、100 ステップ・LLM なしの場合に 3 秒以内を目標とする。 |
| **NF-03** | LLM 呼び出しは非同期で実行し、完了後にトースト通知で知らせる。 |
| **NF-04** | Azure AI Foundry の API キーおよびエンドポイント URL は Windows DPAPI（`ProtectedData`）で暗号化して config.json に保存する。平文では保存しない。 |
| **NF-05** | UIA API 呼び出しは撮影処理と同一の `Task.Run` スレッド内で実行し、1 呼び出しあたり 200ms のタイムアウトを設ける。タイムアウト時は OCR にフォールバックする。 |
| **NF-06** | OCR 処理には `Windows.Media.Ocr`（WinRT API、.NET 8 から利用可能）を優先的に使用し、利用不可の場合は Tesseract（NuGet: `Tesseract`）にフォールバックする。 |
| **NF-07** | docx 生成ライブラリは `DocumentFormat.OpenXml`（NuGet: `DocumentFormat.OpenXml`）を使用する。 |

## 7. 実装ロードマップ

### フェーズ 1（基盤）
- セッション管理 (S-01〜S-05)
- ルールベースのステップ記録 (E-01〜E-05)
- Markdown 出力 (O-01, O-04〜O-06)
- 基本設定 UI (C: Enabled / OutputFolder / ScreenshotMode / KeyboardMode)

### フェーズ 2（UI 情報取得）
- UIA 連携 (U-01, U-04, U-05)
- OCR フォールバック (U-02, U-03)
- 要レビューマーク (U-03)

### フェーズ 3（Word 出力）
- docx 生成・書式 (O-02, O-03)
- 表紙・目次・ヘッダー/フッター (D-01, D-07, O-03)
- ユーザーテンプレート対応 (O-07)

### フェーズ 4（LLM 連携）
- API キー管理・DPAPI 暗号化 (L-01, L-02, NF-04)
- 操作目的テキスト LLM 改善 (L-03)
- ダイジェスト生成 (L-04)
- モデル設定 (L-07)

## 8. 既存機能との関係

| 既存機能 | 手順書機能との関係 |
|---|---|
| TriggerOrchestrator | ステップ記録のトリガー供給元。FireCapture() 完了後に ManualSessionRecorder へ通知する。 |
| TriggerEvent | ManualStep の基礎データとして再利用する。 |
| FileStorage | スクリーンショットの保存先パスを ManualStep.ImagePath に設定する。 |
| MetadataLogger | JSONL ログと手順書は独立して共存する（相互参照なし）。 |
| NotifyIconWrapper | トレイメニューに新項目を追加する。 |
| ConfigStore / AppConfig | ManualGenConfig サブクラスを AppConfig に追加する。 |
