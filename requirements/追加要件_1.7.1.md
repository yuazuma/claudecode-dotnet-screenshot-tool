## 1. FR-I1: MP4 エクスポートの Azure Windows Server 2025 対応

### 1.1 背景・課題

Azure Windows Server 2025 環境では **Media Foundation の H.264 エンコーダー MFT が存在しない**
（Desktop Experience・Media Feature Pack 未インストール）。
その結果、IMFSinkWriter.Finalize() が `HRESULT=0xC00D4A44` で失敗し、MP4 エクスポートが
AVI（MJPEG）フォールバックへ移行していた。

要件: **MP4（H.264）形式のみを出力し、AVI フォールバックは廃止する。**

### 1.2 実装方針

H.264 エンコーダーの取得を 3 段階で試みる。いずれも失敗した場合のみエラーとし、
AVI への退避は行わない。

| 優先度 | クラス | 方式 |
|---|---|---|
| 1 | `MfVideoWriter` | IMFSinkWriter H.264（従来方式） |
| 2 | `H264Mp4Writer` | H.264 MFT 直接呼び出し（MPEG-4 マルチプレクサーバイパス） |
| 3 | `FfmpegMp4Writer` | FFmpeg subprocess（PATH または既知インストールパスを自動検索） |

### 1.3 H264Mp4Writer（FR-I1-A）

| 項目 | 内容 |
|---|---|
| 目的 | IMFSinkWriter の MPEG-4 マルチプレクサー（muxer）をバイパスし、H.264 MFT エンコーダーのみを直接使用する |
| MFT 検索 | `MFTEnum(MFT_CATEGORY_VIDEO_ENCODER, MFT_ENUM_FLAG_SYNCMFT, ...)` で H.264 エンコーダー CLSID を列挙 |
| コンテナ | ISO BMFF (MP4) を自前実装で書き出し（`ftyp` / `mdat` / `moov` / `avc1` / `avcC`） |
| SPS/PPS | encoder 出力の NAL unit から type 7（SPS）・8（PPS）を抽出して `avcC` ボックスに埋め込む |
| Annex-B → AVCC | スタートコード形式 NAL を length-prefix 形式（AVCC）に変換してから mdat に書き込む |
| 備考 | H.264 MFT が存在しない環境では `COMException` をスローし、上位で `FfmpegMp4Writer` へフォールバック |

### 1.4 FfmpegMp4Writer（FR-I1-B）

| 項目 | 内容 |
|---|---|
| 目的 | FFmpeg の libx264 エンコーダーを使用し、H.264 MFT なしで MP4 を生成する |
| フレーム保存 | 各フレームを JPEG（品質 92）として一時フォルダに書き出す（ARGB→24bppRgb 変換を経てから保存） |
| concat ファイル | `ffconcat version 1.0` 形式でフレームリストと各フレームの表示秒数を記述 |
| FFmpeg 検索順 | ① アプリフォルダ直下 ② `tools/ffmpeg.exe` ③ 既知インストールパス ④ PATH 環境変数 |
| FFmpeg コマンド | `ffmpeg -f concat -safe 0 -i concat.txt -c:v libx264 -pix_fmt yuv420p -movflags +faststart output.mp4` |
| エラー時 | 終了コード ≠ 0 の場合は stderr を含む例外をスローし、エラーログに出力 |
| FFmpeg 未検出 | `FileNotFoundException` をスロー。メッセージに Media Feature Pack インストールコマンドを案内 |

### 1.5 AVI フォールバック廃止

`VideoGenerator.GenerateForSteps()` から `MjpegAviWriter` の呼び出しを削除する。
Step 2c（FfmpegMp4Writer）も失敗した場合は MP4 生成失敗としてエラーログを出力し、
AVI ファイルは生成しない。

`MjpegAviWriter.cs` はソースとして残すが、VideoGenerator からは参照しない。

### 1.6 VideoGenerator バグ修正（付随）

`CalcFrameDuration()` の戻り値は**秒単位**であるにもかかわらず、
Step 2b の `H264Mp4Writer` 呼び出し箇所で `dur / 10_000_000.0` という誤った除算が
行われていた。`AddVideoFrame(bitmaps[0], dur)` に修正する。

### 1.7 FFmpeg のインストール要件

| 環境 | 対応方法 |
|---|---|
| Windows クライアント（H.264 MFT 搭載） | Step 1（MfVideoWriter）で成功 → FFmpeg 不要 |
| Azure Windows Server（H.264 MFT なし） | Step 3 が動作するには FFmpeg が必要 |
| FFmpeg インストール方法 | `winget install Gyan.FFmpeg` または DISM Media Feature Pack |

### 1.8 非機能要件

| ID | 要件 |
|---|---|
| NF-I101 | 3 段階すべて失敗した場合は AVI を生成せず、エラーログに FFmpeg / Media Feature Pack のインストール方法を案内する |
| NF-I102 | 一時フォルダ（JPEG フレーム）は `FfmpegMp4Writer.Dispose()` 時に削除する |
| NF-I103 | JPEG 保存時は ARGB → Format24bppRgb 変換を行い `ArgumentException` を回避する |
| NF-I104 | 動画フレームの横幅・高さはいずれも偶数に切り捨てる（H.264 要件） |

## 2. FR-I2: ヘッドレスエクスポート CLI フラグ

### 2.1 概要

GUI なしで特定プロジェクトをエクスポートする **`--export` CLI フラグ**を追加する。
RDP 環境での動作確認や CI / バッチ実行に利用できる。

### 2.2 コマンドライン書式

```
AutoScreenshot.exe --export <プロジェクトフォルダパス> [--type <種別リスト>]
```

| 引数 | 必須 | 説明 |
|---|---|---|
| `--export <path>` | 必須 | `.ascproj` フォルダの絶対パス（または相対パス） |
| `--type <types>` | 省略可 | カンマ区切りで `md`, `html`, `video`, `images`, `zip` を指定。省略時はすべて実行 |

### 2.3 終了コード

| コード | 意味 |
|---|---|
| 0 | 全エクスポート成功 |
| 1 | 一部失敗（少なくとも 1 つのエクスポートでエラー発生） |
| 2 | 引数エラー（パス未指定・プロジェクト読み込み失敗） |

### 2.4 実装クラス

**`Services/ExportCliRunner.cs`**（新規）

- `App.xaml.cs` の `OnStartup()` で `--export` フラグを検出したら `RunAsync()` を呼び出し、
  結果の終了コードで `Environment.Exit()` する
- GUI ウィンドウは一切表示しない（トレイアイコン含む）
- Serilog のログは既存のファイルログに出力される

### 2.5 非機能要件

| ID | 要件 |
|---|---|
| NF-I201 | `--export` フラグが存在する場合、GUI を表示しない |
| NF-I202 | エクスポート完了後に `Environment.Exit(exitCode)` を呼び出してプロセスを終了する |
| NF-I203 | 設定ファイル（`config.json`）は通常起動時と同じパスから読み込む |

## 3. FR-I3: Markdown 手順書の画像パス修正と最大横幅制限

### 3.1 背景・課題

v1.7.0 以前の `MarkdownManualWriter` は、画像パスを
`../images/step_001.png` のように **親ディレクトリを参照する相対パス** で出力していた。

VS Code・GitHub・多くのブラウザはセキュリティ上の理由から `..` を含むパスを
拒否するため、**画像が表示されない**問題が発生していた。

また、高解像度ディスプレイで撮影した画像（例: 1512×948 px）をそのままコピーすると
Markdown 文書内で表示が大きくなりすぎる課題があった。

### 3.2 画像コピー先の変更

Markdown ファイルと同じ `exports/` フォルダ内に **`{mdファイル名}_images/` サブフォルダ** を作成し、
すべての参照画像をそこにコピーする。

| 変更前 | 変更後 |
|---|---|
| `../images/step_001.png` | `20260530_143022_操作手順書_images/step_001.png` |

#### 効果

- `..` なしの同一階層相対パスになるため、VS Code・GitHub 等で画像が正常表示される
- アノテーション済み一時ファイルもコピーされるため、一時ファイル削除後も参照が有効

### 3.3 1200px 横幅リサイズ（FR-I3-B）

画像コピー時に横幅を確認し、**1200px を超える場合はアスペクト比を維持して 1200px にリサイズ**する。

| 条件 | 動作 |
|---|---|
| 横幅 > 1200px | HighQualityBicubic でリサイズしてから保存 |
| 横幅 ≤ 1200px | そのままバイトコピー（無劣化） |
| リサイズ失敗時 | ログに警告を出力し、元ファイルをそのままコピー（フォールバック） |

### 3.4 重複コピー防止（FR-I3-C）

before/after が同じ画像ファイルを参照している場合（before 画像なし → フォールバックで after と同一になる）、
ソースパスをキャッシュして同一ファイルの重複コピーを防止する。

### 3.5 影響クラス

| クラス | 変更内容 |
|---|---|
| `Services/MarkdownManualWriter.cs` | `CopyImage()` メソッド追加（画像コピー・リサイズ・キャッシュ）、`WriteImage()` メソッド追加（リサイズロジック） |

### 3.6 非機能要件

| ID | 要件 |
|---|---|
| NF-I301 | `_images/` フォルダは MD ファイルと同じ `exports/` ディレクトリに作成する |
| NF-I302 | ファイル名衝突時は連番サフィックス（`_1`, `_2`, ...）を付与して回避する |
| NF-I303 | WebP 形式は GDI+ 非対応のため、リサイズ時は PNG で代替保存する |
| NF-I304 | ARGB ビットマップは JPEG 保存前に Format24bppRgb に変換する |

## 4. FR-I4: RDP セッションでのスクリーンキャプチャ改善

### 4.1 背景・課題

Azure Windows Server 2025 へ RDP 接続した状態で AutoScreenshot を使用すると、
`Graphics.CopyFromScreen()` が **デスクトップ壁紙の単色背景しか取得できない**問題が発生した。

原因: `CopyFromScreen()` は GDI の `BitBlt` を使用するが、RDP セッション下では
DWM（Desktop Window Manager）コンポジターの出力が GDI レイヤーに反映されない。

### 4.2 解決方針

RDP セッション検出時は **Windows.Graphics.Capture (WGC) API** を使用する。
WGC は DWM コンポジターの出力を直接読み取るため、RDP でも実際のウィンドウ内容を取得できる。

| 方法 | RDP での動作 | 備考 |
|---|---|---|
| `GDI CopyFromScreen` | ✗ 壁紙色のみ（RGB 24,24,24 の単色） | distinct colors ≈ 23 |
| `BitBlt + CAPTUREBLT` | ✗ 同上 | 同様の GDI 制限 |
| **Windows.Graphics.Capture** | ✅ 実際の画面内容を取得 | distinct colors ≈ 42、タスクバー等を確認 |

### 4.3 実装（FR-I4-A: WgcCapture）

**`Services/WgcCapture.cs`**（新規）

| ステップ | 内容 |
|---|---|
| 1 | `MonitorFromPoint()` で HMONITOR を取得 |
| 2 | `RoGetActivationFactory` + `IGraphicsCaptureItemInterop::CreateForMonitor` で `GraphicsCaptureItem` を生成 |
| 3 | `D3D11CreateDevice`（ハードウェア優先 → WARP フォールバック）で D3D11 デバイスを生成 |
| 4 | `CreateDirect3D11DeviceFromDXGIDevice` で WinRT `IDirect3DDevice` ラッパーを取得 |
| 5 | `Direct3D11CaptureFramePool.CreateFreeThreaded` でフレームプールを生成（スレッドプールから安全） |
| 6 | `TrySuppressBorder()` で黄色キャプチャ通知ボーダーを非表示（Windows 11+） |
| 7 | `session.StartCapture()` 後、`FrameArrived` で 1 フレームを取得（最大 3 秒タイムアウト） |
| 8 | `SoftwareBitmap.CreateCopyFromSurfaceAsync` で D3D サーフェスを CPU アクセス可能なソフトウェアビットマップに変換 |
| 9 | `BitmapEncoder`（BMP 形式）でエンコードして `System.Drawing.Bitmap` に変換 |

#### ポイント: `CreateFreeThreaded` の使用

`Direct3D11CaptureFramePool.Create`（ディスパッチャースレッド必須）ではなく
`CreateFreeThreaded`（スレッドプール・任意スレッドから安全）を使用する。
TriggerOrchestrator のキャプチャは `Task.Run` でスレッドプール上で実行されるため、
`Create` を使うと `FrameArrived` が永遠に発火しないというバグがある。

#### ポイント: 黄色ボーダー非表示（FR-I4-B）

`GraphicsCaptureSession.IsBorderRequired` は Windows 11 (Build 22000) 以降で追加されたが、
TFM 17763 では型定義に存在しない。次の 2 段階フォールバックで設定する：

1. `(object)session is IGraphicsCaptureSession3 s3` → `s3.put_IsBorderRequired(false)`
   （`IGraphicsCaptureSession3`: GUID `BFEE7A93-F3BD-4A41-A011-2F5A28E62735`、`InterfaceIsIInspectable`）
2. `IWinRTObject.NativeObject.ThisPtr` + `Marshal.QueryInterface` → vtable[7] を unsafe 直接呼び出し

### 4.4 CaptureService の変更（FR-I4-C）

| 変更点 | 内容 |
|---|---|
| RDP 検出 | `SystemInformation.TerminalServerSession` を static readonly フィールドでキャッシュ |
| WGC 可否 | `WgcCapture.IsSupported()` を static readonly フィールドでキャッシュ |
| 優先フロー | RDP セッション かつ WGC サポート → WGC 試行 → 失敗時 GDI フォールバック |

### 4.5 非機能要件

| ID | 要件 |
|---|---|
| NF-I401 | WGC キャプチャが失敗した場合は `Log.Warning` を出力し、GDI `CopyFromScreen` にフォールバックする |
| NF-I402 | WGC フォールバックは RDP 検出 + WGC サポート確認を両方満たした場合のみ試行する |
| NF-I403 | D3D11 デバイス生成はハードウェアを優先し、失敗時は WARP（ソフトウェアレンダラー）を使用する |
| NF-I404 | WGC フレーム取得タイムアウトは 3 秒とする |
| NF-I405 | 黄色ボーダー非表示の設定失敗は無視する（非対応 OS では try-catch で握り潰す） |
| NF-I406 | WGC 実装に新規 NuGet パッケージを追加しない（.NET 8 の組み込み WinRT サポートと COM P/Invoke のみ使用） |

## 5. BF-I1 ～ BF-I4: バグ修正

### BF-I1: JSON プロパティ名重複

| 項目 | 内容 |
|---|---|
| 対象ファイル | `Models/AppConfig.cs`（`ManualGenConfig` クラス） |
| 症状 | `OutputFolder` と `LegacyOutputFolder` がともに `"outputFolder"` でシリアライズされ、`config.json` 読み込み時に `JsonException` が発生 |
| 修正方法 | `OutputFolder` フィールドを削除し、`VideoOutputFolder` に統一 |

### BF-I2: FolderTemplate で `.ascproj` の `s` が秒指定子として展開される

| 項目 | 内容 |
|---|---|
| 対象ファイル | `Services/FolderTemplateService.cs` |
| 症状 | `.ascproj` 拡張子の小文字 `s` が `DateTime.ToString()` の秒（`s`）書式指定子として誤展開される（例: `20260530_143022_title.a12cproj`） |
| 修正方法 | プレースホルダー展開後の値を `\x02...\x03` でエスケープし、`DateTime.ToString()` 適用後にエスケープを解除する（`ProtectFromDateTime()` / `UnprotectFromDateTime()` メソッド追加） |

### BF-I3: FfmpegMp4Writer の JPEG 保存 ArgumentException

| 項目 | 内容 |
|---|---|
| 対象ファイル | `Services/FfmpegMp4Writer.cs` |
| 症状 | ARGB（32bpp アルファ付き）ビットマップを `ImageCodecInfo`（JPEG）で保存しようとすると `System.ArgumentException: Parameter is not valid.` が発生 |
| 修正方法 | 保存前に `Format24bppRgb`（アルファなし 24bpp）に `Graphics.DrawImage` で変換してから保存 |

### BF-I4: MarkdownManualWriter の MemoryStream 解放後 Bitmap 保存エラー

| 項目 | 内容 |
|---|---|
| 対象ファイル | `Services/MarkdownManualWriter.cs`（`SoftBitmapToGdiAsync` ではなく `CopyImage` 周辺） |
| 症状 | `new Bitmap(memoryStream)` で作成した Bitmap は MemoryStream への参照を保持するため、`using var ms = ...` でストリームを閉じると後続の `Save()` 呼び出しで `ExternalException: A generic error occurred in GDI+.` が発生 |
| 修正方法 | `new Bitmap(tmp)` で stream 非依存の独立コピーを作成してから return する |
